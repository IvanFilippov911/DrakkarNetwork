# C4 Level 2 — Container View

Документ показывает Drakkar Network на уровне контейнеров: какие приложения, сервисы и хранилища участвуют в работе системы.

![C4 Container View](./assets/c4-container.svg)

## Назначение диаграммы

C4 Level 2 раскрывает внутреннюю структуру Drakkar Network после системного контекста.

На этом уровне видно разделение системы на несколько зон:

- пользовательский слой;
- административный слой;
- backend/control-plane слой;
- VPN node слой;
- слой хранения данных;
- observability слой.

## Основные контейнеры

### Nginx Reverse Proxy

Входная точка для HTTP(S)-трафика.

Отвечает за:

- маршрутизацию запросов;
- TLS termination;
- проксирование запросов к User API, Admin API и frontend-приложениям.

### User WebApp

Пользовательский frontend.

Используется для получения VPN-доступа, отображения статуса и выдачи конфигурации пользователю.

### Admin WebApp

Административный frontend.

Используется для управления серверами, peer'ами, transport profiles, ошибками, метриками и operational state.

### User API

Backend для пользовательских сценариев.

Основные задачи:

- проверка доступа пользователя;
- работа с устройствами;
- получение текущей VPN-конфигурации;
- запуск peer provisioning flow.

### Admin API

Backend для административных сценариев.

Основные задачи:

- управление серверами;
- управление transport profiles;
- управление VPN peer'ами;
- просмотр ошибок, метрик и состояния инфраструктуры;
- запуск control-plane операций, например применения transport profile.

### Telegram Bot

Точка входа пользователя в сервис.

Используется для перехода в WebApp и базового Telegram flow.  
В рамках showcase не является главным инженерным фокусом.

### PostgreSQL

Основное хранилище состояния системы.

Хранит:

- пользователей и устройства;
- VPN peer'ы;
- серверы;
- transport profiles и activations;
- jobs;
- ошибки и operational events.

### Redis

Используется для временных и инфраструктурных сценариев:

- кэш;
- runtime state;
- агрегированные метрики;
- coordination scenarios.

### VPN Node Agent

Сервис на VPN node.

Принимает команды от control plane и применяет изменения к локальному VPN runtime.

Основные задачи:

- создать или обновить peer;
- применить transport profile;
- получить состояние node;
- отдать runtime-метрики.

### Xray Runtime

VPN runtime на node.

Обслуживает реальные VPN-подключения пользователей.

### Agent SQLite State

Локальное состояние node agent.

Используется для хранения данных, которые нужны agent'у рядом с runtime.

### Observability Stack

Слой наблюдаемости.

Включает:

- Prometheus;
- Grafana;
- Loki;
- Promtail.

Используется для метрик, логов, dashboard'ов и диагностики.

## Ключевые взаимодействия

Диаграмма показывает четыре основные группы взаимодействий.

### Пользовательский контур

Пользователь входит через Telegram Bot или User WebApp.  
User API отвечает за пользовательские сценарии: проверку доступа, работу с устройствами и выдачу VPN-конфигурации.

### Административный контур

Администратор работает через Admin WebApp.  
Admin API управляет серверами, peer'ами, transport profiles, ошибками, метриками и operational state.

### Контур управления VPN nodes

Control plane не применяет изменения к Xray Runtime напрямую.  
Для этого используется VPN Node Agent, установленный рядом с Xray на конкретном сервере.

Через agent выполняются операции вроде:

- создания или обновления peer'а;
- применения transport profile;
- получения состояния node;
- сбора runtime-метрик.

### Контур наблюдаемости

API, workers и agents отдают логи, метрики и operational events в observability stack.

Этот контур нужен не только для dashboard'ов, но и для диагностики проблем: недоступности серверов, ошибок provisioning, ошибок применения transport profile и сетевых деградаций.
